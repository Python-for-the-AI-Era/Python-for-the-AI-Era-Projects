# Project 06: Robust File Processing CLI

### Overview
This project targets production engineering patterns. True Python software must run reliably in hostile environments. You will implement custom class exception structures, design a multi-stream log pipeline with a cascading rotating target footprint, emit system alerts via standard library warning controls, and craft a self-recording batch context manager.

---

### The Architecture Contract

Your implementation must strictly satisfy these enterprise criteria:

#### 1. Explicit Domain Exception Slicing
Never raise generic built-in runtime exceptions like `ValueError` or `Exception` for custom application logic.
* You must construct an isolated object graph branching from `AppError`.
* Every custom exception must explicitly bind the targeting filesystem `Path` object along with a standardized uppercase logging tag string (`code`), such as `"FILE_TOO_LARGE"`. This guarantees upstream collectors can instantly categorize problems without parsing random text messages.

#### 2. Dual-Channel Logging Pipeline
Configure your root logging environment immediately at execution startup:
* **Storage Footprint**: Wire a `RotatingFileHandler` writing directly to `app.log`. Set its parameter ceiling bounds to `maxBytes=5_242_880` (5MB) with a rollover threshold of `backupCount=3`.
* **Console Monitoring**: Stack a secondary `StreamHandler` pointing to standard error output. Set its runtime threshold filter explicitly to `logging.WARNING` so trivial internal trace info does not clutter the user's view.

#### 3. Graceful Error Suppression with Context Managers
Your `ProcessingReport` class is a specialized data-recording runtime harness. 
* By structuring the `__exit__(self, exc_type, exc_val, exc_tb)` method signature, your code must inspect incoming exception objects.
* If `exc_val` is an instance of `AppError`, record its specific diagnostic codes and metrics into a summary dictionary, output the data cleanly as JSON to disk, and return `True`. This suppresses the crash, protecting loops from breaking when processing large directories of chaotic input files.

#### 4. Diagnostic Alerts (`warnings.warn`)
For non-breaking issues—like parsing a file that is more than 30 days old or tracking sizes over 20MB—do not throw fatal exceptions. Use `warnings.warn` combined with appropriate classification categories (`UserWarning`) to notify administrators cleanly without stopping execution.

```python
file_size_bytes = path.stat().st_size
last_modified_timestamp = path.stat().st_mtime
```
